# 第 1 章 项目全景

## 1.1 我们要构建什么

一个运维 OnCall 助手，解决两个核心场景：

| 场景 | 问题 | 方案 |
|------|------|------|
| **RAG 问答** | 运维知识散落在文档里，新人查不到 | 文档向量化 + 多轮对话检索 |
| **AIOps 诊断** | 告警来了不知道怎么排查 | 大模型自动制定排查计划，调用工具执行，动态调整 |

## 1.2 服务拓扑

项目由 3 个独立进程组成：

```
┌─────────────────────┐
│   FastAPI (9900)    │  主服务，处理 HTTP 请求
│                     │
│  ┌───────────────┐  │
│  │  RAG Agent    │────── Milvus (19530)  向量检索
│  └───────────────┘  │
│                     │
│  ┌───────────────┐  │
│  │  AIOps Agent  │──┼── MCP Client
│  └───────────────┘  │       │
└─────────────────────┘       │
              ┌───────────────┼───────────────┐
              ▼                               ▼
   ┌──────────────────┐          ┌──────────────────┐
   │ CLS MCP (8003)   │          │ Monitor MCP(8004)│
   │ 日志查询工具      │          │ 监控数据工具      │
   └──────────────────┘          └──────────────────┘
```

**关键设计决策**：MCP 服务器独立进程运行，FastAPI 作为 MCP 客户端连接。这样工具服务可以独立部署、升级，不影响主服务。

## 1.3 两条数据流

**RAG 问答流**：
```
用户提问 → API 层 → RagAgentService → ChatQwen + 工具
                                      ├── retrieve_knowledge → Milvus 检索
                                      ├── get_current_time
                                      └── MCP 工具（日志/监控）
                                   → 流式返回答案
```

**AIOps 诊断流**：
```
用户发起诊断 → API 层 → AIOpsService → LangGraph StateGraph
                                         │
                    ┌── Planner ←──┐     │
                    │  制定计划      │     │
                    └── Executor ←──┘    │
                       调用 MCP 工具      │
                    ┌── Replanner ←─┘    │
                    │  评估+调整           │
                    └──→ 继续执行 or 出报告
```
